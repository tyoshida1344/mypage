# Review Perf — パフォーマンスレビュー

変更差分を「コード量を減らせないか」「処理を軽くできないか」の2軸で確認する。

## 対象の特定

1. 変更ファイルを取得する:

```sh
git diff main --name-only
```

2. 変更ファイルの全体を読む（diff だけでなく前後の文脈が必要）
3. 変更ファイルを参照・利用している関連ファイルも特定して読む（同じ CSS 変数を使うファイル、import 元のコンポーネントなど）

## 確認観点

以下の観点で変更箇所とその周辺を確認する。全てを機械的にチェックするのではなく、**実際に効果がある指摘** に絞る。

### コード量の削減

- 既存のコンポーネント（`SectionWrapper` 等）や Nuxt の auto-import で置き換えられる独自実装がないか
- 重複するロジックを composable やユーティリティに抽出できないか
- 不要な変数・関数・条件分岐がないか
- 既存の CSS 変数（`assets/css/main.css`）で代用できるハードコードされた値がないか
- テンプレート内の冗長な記述を `v-for` や computed で簡潔にできないか

### 処理の軽量化

- `computed` で済むものを `watch` や `methods` で書いていないか
- `v-if` と `v-show` の使い分けが適切か（頻繁な切り替えは `v-show`、レアケースは `v-if`）
- 不要なリアクティビティ（`ref` / `reactive`）を使っていないか（静的データは plain object で十分）
- ループ内の不変な計算をループ外や computed に出せないか
- 画像やフォントの読み込みが最適化されているか（lazy loading、プリロード）
- CSS で実現できるアニメーション・レイアウトを JS で書いていないか

## 出力

指摘事項を以下の形式で報告する:

```
### パフォーマンスレビュー結果

**コード量削減**
- [ ] `ファイル名:行番号` — 指摘内容（改善案）

**処理軽量化**
- [ ] `ファイル名:行番号` — 指摘内容（改善案）

指摘なしの場合: 「パフォーマンス上の問題は見つかりませんでした」
```

修正が求められている場合（issue-handler 経由）は、報告後に指摘箇所を修正する。
